## Bugs found and fixed

### Bug 1 — `print_report()` missing → `AttributeError` crash
**File:** `core/memory_manager.py`

`src/core/memory_aware_patch.py` calls `scheduler.print_report()` but `MemoryAwareJobScheduler` had no such method. This would crash immediately when the patch is applied.

**Fix:** Added `print_report()` method to `MemoryAwareJobScheduler`.

---

### Bug 2 — `_estimate_memory()` ignores vibronic modes → OOM (root cause)
**Files:** `core/memory_manager.py`, `core/hops_simulator.py`

`MemoryAwareJobScheduler._estimate_memory()` only scaled with `L` and `K`, not with the number of hierarchy modes. For the full FMO run (189 modes vs 21 in the audit), the estimate stayed at 6 GB/traj instead of ~54 GB/traj, yielding `n_jobs=10` → ~540 GB attempted on 128 GB.

**Fix:** Added `modes_factor = n_hierarchy_modes / 21.0` scaling:

```python
# before
def _estimate_memory(self) -> float:
    estimate = BASE_TRAJ_MEMORY_GB * l_factor * k_factor

# after
def _estimate_memory(self, n_hierarchy_modes: int = 21) -> float:
    modes_factor = max(1.0, n_hierarchy_modes / 21.0)
    estimate = BASE_TRAJ_MEMORY_GB * l_factor * k_factor * modes_factor
```

Updated `validate_and_adapt()` signature to accept and forward `n_hierarchy_modes`.  
Updated call site in `core/hops_simulator.py` to pass `len(gw_sysbath)`.

**Effect on full FMO run:**
```
modes_factor = 189 / 21 = 9.0
traj_mem_gb  = 6.0 × 1.0 × 1.0 × 9.0 = 54 GB/traj
n_mem_jobs   = floor(80 GB / 54 GB) = 1
n_jobs       = min(1, 48, 10) = 1   ← sequential, safe
```

---

### Bug 3 — `src/core/hops_simulator.py` not updated (260511 fixes not ported)
**File:** `src/core/hops_simulator.py`

This file was still the original 260511 version. The `modes_factor` fix and `MAX_N_JOBS` cap from the 260511 session were never applied here.

**Fix:** Applied the same two changes as in 260511:
- Replaced `_get_memory_estimate()` with the `modes_factor` version
- Changed `n_jobs = min(n_mem_jobs, cpu_count)` → `min(n_mem_jobs, cpu_count, MAX_N_JOBS)`
- Passed `n_hierarchy_modes` from `self.system_param["GW_SYSBATH"]` at the call site

---

### Bug 4 — `MAX_N_JOBS` constant missing
**Files:** `core/constants.py`, `src/core/constants.py`

Neither constants file defined `MAX_N_JOBS`, so the import in `src/core/hops_simulator.py` would fail.

**Fix:** Added to both files:
```python
MAX_N_JOBS: Final[int] = 10  # Hard cap on parallel trajectories to prevent OOM
```
Added `MAX_N_JOBS` to both `try/except` import blocks in `src/core/hops_simulator.py`.

## Full OOM Audit — 2026-05-12 (post-260512 creation)

Server constraints: 128 GB RAM, 48 logical CPUs.  
Full FMO run: 189 hierarchy modes, L=8, K=2 → **~54 GB/traj**.  
Safe maximum: `floor(128 × 0.66 / 54) = 1` concurrent trajectory.

### Critical bugs fixed

| # | File | Bug | Fix |
|---|---|---|---|
| 1 | `core/constants.py`, `src/core/constants.py` | `MAX_N_JOBS=10` → 10×54=540 GB OOM | `MAX_N_JOBS=1` |
| 2 | `core/hops_simulator.py` | `len(gw_sysbath)` NameError (out of scope) → except fallback `n_jobs=min(100,48)=48` → 2592 GB OOM | `self.system_param.get("GW_SYSBATH",[])` + fallback `n_jobs=1` |
| 3 | `utils/parallel_utils.py` | `get_safe_n_jobs(default=2.0)` → n_jobs=42 | default `54.0` |
| 4 | `utils/parallel_utils.py` | `parallel_trajectory_simulation(n_workers=40)` hardcoded | `n_workers=1` |
| 5 | `core/memory_manager.py` | `validate_and_adapt(default=21)` → 6 GB estimate → n_jobs=14 | default `189` |
| 6 | `core/memory_manager.py` | `_estimate_memory(default=21)` same issue | default `189` |
| 7 | `core/memory_manager.py` | `batch_size=max(n_jobs, n_traj//4)=25` → 25 concurrent objects | `batch_size=n_jobs` |
| 8 | `src/core/memory_aware_patch.py` | `validate_and_adapt()` called without n_hierarchy_modes → 6 GB estimate | pass `len(self.system_param["GW_SYSBATH"])` |
| 9 | `reproducibility/main.py` (temp sweep) | `get_safe_n_jobs(BASE_TRAJ_MEMORY_GB=6.0)` → n_jobs=14 | `get_safe_n_jobs(54.0)` |
| 10 | `reproducibility/main.py` (disorder) | same | `get_safe_n_jobs(54.0)` |
