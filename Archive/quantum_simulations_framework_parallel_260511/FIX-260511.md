### Problem
All existing tests and parameters were tuned for a 4-core / 16 GB laptop. Running them on the server wasted available resources and did not exercise production-scale parameters.

### Files created / modified

Full production config for the ThinkStation P720.

| Parameter | Laptop | Server |
|---|---|---|
| `L_max` | 3 | 8 |
| `K` (Matsubara) | 2 | 2 (unchanged — K > 2 → OOM even on 128 GB) |
| `n_traj` | 4 | 100 |
| `n_traj_temp_sweep` | 2 | 10 |
| `n_disorder_samples` | 5 | 100 |
| `time_max` | 200 fs | 1000 fs |

| `sbd_bundles_per_site` | 2 | 6 |
| `n_workers` | — | 40 |
| `dpi` | 300 | 600 |

Note: `K` stays at 2 on both laptop and server. K=10 would generate ~10¹¹ hierarchy states (77 modes × K=10), causing OOM even on 128 GB. K=2 is physically converged at T=295 K because ν₁ ≈ 1300 cm⁻¹ >> γ_D = 50 cm⁻¹.

#### `tests/test_server_suite.py` *(new)*
9 server-scale tests marked `@pytest.mark.server`:
1. Hamiltonian properties (Hermiticity, shape, real eigenvalues)

Run with:

pytest tests/test_server_suite.py -m server -v -n 40   # with pytest-xdist

## 2. `run_cluster_gpu.sh` — 5 bugs fixed

**File:** `run_cluster_gpu.sh` (project root)

### Bug 1 — Missing working-directory anchor *(critical)*
`MAIN_SCRIPT` was a bare relative path. Running the script from any directory other than `Redac_Paper1/` caused an immediate `No such file or directory` crash.

**Fix:** anchored all paths to `SCRIPT_DIR` using `$(dirname "${BASH_SOURCE[0]}")`.

### Bug 2 — Log file written to random CWD
`LOG_FILE="reproducibility_cluster.log"` landed wherever the script was launched from, not inside the framework log directory.

**Fix:** changed to `$FRAMEWORK_DIR/reproducibility/logs/cluster_gpu_<timestamp>.log`.

### Bug 3 — `--config` never forwarded
No mechanism to pass `--config config/server_parameters.yaml` to `main.py`.

**Fix:** forwarded `$*` to `main.py`. Usage:
```bash
bash run_cluster_gpu.sh --config config/server_parameters.yaml
```

### Bug 4 — `XLA_PYTHON_CLIENT_MEM_FRACTION` contradicted `parallel_config.yaml`
Script set `0.33`; `parallel_config.yaml` specifies `gpu_memory_fraction: 0.9`.

**Fix:** aligned to `0.90`.

### Bug 5 — PID not persisted
PID was only printed to the terminal; untrackable after SSH session close.

**Fix:** PID written to `$LOG_DIR/cluster_gpu.pid`. Kill command printed at launch:
```bash
kill $(cat reproducibility/logs/cluster_gpu.pid)
```

## 3. `n_jobs` hard cap — OOM fix (first pass)

### Problem
The memory-aware `n_jobs` formula yielded 13 parallel trajectories for the full FMO run (189 hierarchy modes), causing OOM on 128 GB.

### Files modified
- `core/constants.py`
- `src/core/constants.py`
- `core/hops_simulator.py`
- `src/core/hops_simulator.py`

### Change
Added `MAX_N_JOBS: Final[int] = 10` to both constants files.

Applied hard cap at the `n_jobs` assignment:
```python
# before
n_jobs = min(n_mem_jobs, cpu_count)
# after
n_jobs = min(n_mem_jobs, cpu_count, MAX_N_JOBS)
```

## 4. `_get_memory_estimate` — OOM root-cause fix (second pass)

### Problem (diagnosed from `cluster_gpu_20260511_160723.log`)
The memory estimate was **wrong for the full FMO run**. The formula only scaled with `L` and `K`, ignoring the number of hierarchy modes entirely.

| Run | Modes | Est/Traj | n_jobs | Result |
|---|---|---|---|---|
| Audit (L=6/7/8, no vibronic) | 21 | 3.4–6.0 GB | 10 | ✅ OK |
| **Full FMO (Step 3)** | **189** | **6.0 GB** | **10** | 💥 OOM |

With 189 modes the real cost is ~54 GB/traj. Running 10 in parallel requires ~540 GB → OOM on 128 GB.

### Root cause
`_get_memory_estimate()` used `BASE_TRAJ_MEMORY_GB × (L/8)² × (K/2)` but never accounted for `n_hierarchy_modes`. The vibronic modes (24 pairs × 7 sites = 168 extra modes) inflate the hierarchy state space ~9×.

### Files modified
- `core/hops_simulator.py`
- `src/core/hops_simulator.py`

### Change

**`_get_memory_estimate()`** — added `n_hierarchy_modes` parameter with `modes_factor` scaling:
```python
# before
def _get_memory_estimate(self) -> float:
    estimate = BASE_TRAJ_MEMORY_GB * l_factor * k_factor

# after
def _get_memory_estimate(self, n_hierarchy_modes: int = 21) -> float:
    n_modes_ref = 21.0  # 3 DL × 7 sites (audit baseline)
    modes_factor = max(1.0, n_hierarchy_modes / n_modes_ref)
    estimate = BASE_TRAJ_MEMORY_GB * l_factor * k_factor * modes_factor
```

**Call site** — passes actual mode count from `system_param`:
```python
# before
traj_mem_gb = self._get_memory_estimate()

# after
n_hierarchy_modes = len(self.system_param.get("GW_SYSBATH", [])) \
    if hasattr(self, "system_param") and self.system_param else 21
traj_mem_gb = self._get_memory_estimate(n_hierarchy_modes)
```

### Effect on full FMO run
```
modes_factor = 189 / 21 = 9.0
traj_mem_gb  = 6.0 × 1.0 × 1.0 × 9.0 = 54 GB/traj
n_mem_jobs   = floor(80 GB / 54 GB) = 1
n_jobs       = min(1, 48, 10) = 1   ← sequential, safe
```

The 100-trajectory ensemble runs sequentially (1 job at a time). No OOM. To allow 2 parallel jobs after profiling actual peak memory, lower `BASE_TRAJ_MEMORY_GB` from `6.0` to `4.0` in `constants.py`.
